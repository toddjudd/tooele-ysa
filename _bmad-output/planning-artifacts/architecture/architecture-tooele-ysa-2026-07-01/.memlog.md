- (direction) Fast path selected by Todd; stack pre-decided: Next.js App Router + TypeScript + Tailwind CSS + Sanity v3 + next-sanity + Sanity CDN + Vercel auto-deploy on push
- (decision) AD-1 Paradigm: Headless CMS Jamstack with strict content-boundary layers — Content (Sanity owns all dynamic data and images), Presentation (Next.js RSC fetches and renders, never mutates), Delivery (Vercel Edge + Sanity Image CDN). Dependency direction: Presentation → Content API only; Content has zero knowledge of Presentation. Prevents: any Sanity schema changes coupling to Next.js routing, or Next.js components writing to Sanity outside Studio.
- (decision) AD-2 Rendering strategy per route: Gatherings (/gatherings) = SSG (no revalidate export, no CMS dependency). Home (/), About Us (/about), Let's Connect (/connect) = ISR with export const revalidate = 60. No page uses SSR (dynamic rendering). Binds: FR-1, FR-3, FR-5 through FR-17. Prevents: a CMS-dependent page being accidentally set static (stale forever) or a Gatherings page pulling a runtime CMS call.
- (decision) AD-3 Server-only CMS data fetching: all GROQ queries execute exclusively in React Server Components or at build time (generateStaticParams). No useEffect + fetch to Sanity API from the client. Binds: all ISR pages and their data-fetching components. Prevents: SANITY_API_READ_TOKEN leaking to the client bundle, and race conditions from client-side CMS polling. Rule: SANITY_API_READ_TOKEN is server-only env var; NEXT_PUBLIC_SANITY_PROJECT_ID and NEXT_PUBLIC_SANITY_DATASET are the only Sanity vars exposed to the client bundle.
- (decision) AD-4 next-sanity is the sole Sanity integration layer: all GROQ queries, typed client configuration, and live-preview plumbing go through next-sanity. No raw fetch() to https://PROJECT.api.sanity.io. All GROQ queries are centralized in lib/sanity/queries.ts and exported as named constants. Binds: FR-1, FR-9, FR-14. Prevents: two implementers hitting the Sanity API through different mechanisms with divergent caching or error handling.
- (decision) AD-5 Sanity CDN owns all image transformation; next/image is configured with a Sanity image loader via @sanity/image-url. Sanity-sourced images never pass through Vercel Image Optimization (no double-processing, no cost double-dip). Static assets (floor plan, app icons) use next/image with default Vercel optimizer. Binds: FR-1 (hero carousel), FR-9 (leader card photos if present). Prevents: two implementers serving Sanity images through different pipelines with inconsistent crop/format params.
- (decision) AD-6 No auth surface in Next.js: all routes are public. Sanity Studio (/studio route, embedded) uses Sanity-managed auth exclusively — no custom middleware, no session, no JWT on the Next.js layer. Binds: all routes, NFR-8. Prevents: inadvertent auth middleware blocking public pages or a second auth system being bolted on.
- (decision) AD-7 Vercel deployment: main branch auto-deploys to production; PR branches get Vercel preview deployments. No manual deploy steps. Environment variables (SANITY_API_READ_TOKEN, NEXT_PUBLIC_SANITY_PROJECT_ID, NEXT_PUBLIC_SANITY_DATASET, NEXT_PUBLIC_SANITY_STUDIO_URL) are configured in the Vercel project dashboard, not in committed files. Binds: A-10. Prevents: divergent deployment targets or secrets committed to version control.
- (decision) AD-8 Tailwind CSS is the sole styling system; no component library (no shadcn, no MUI, no Radix). All color, typography, and spacing tokens from DESIGN.md are implemented as Tailwind CSS config extensions in tailwind.config.ts. Tailwind v4 config syntax applies. Binds: all UI components, NFR-6. Prevents: two implementers adding shadcn or another design system that would diverge from DESIGN.md tokens.
